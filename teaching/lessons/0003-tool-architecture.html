0003 — Tool Registration & Architecture

How Pi extensions register tools, load prompts, and compose with the runtime

The problem

You've seen how anchors are computed and how edits flow through the pipeline. But how does Pi know about the read/edit/insert tools in the first place? How do the tool descriptions the LLM sees get loaded? How does the UI preview work?

This lesson covers the extension registration layer — the thin but essential membrane between the runtime and the implementation.

The extension entrypoint pattern

Every Pi extension is a default-exported function that receives an ExtensionAPI object. The runtime calls this function once at startup. Here is the entire extensions/core.ts: 

import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";
import { registerEditTool } from "../src/edit";
import { registerReadTool } from "../src/read";

export default function (pi: ExtensionAPI): void {
  registerEditTool(pi);
  registerReadTool(pi);

  if (process.env.PI_HASHLINE_DEBUG === "1" || process.env.PI_HASHLINE_DEBUG === "true") {
    pi.on("session_start", async (_event, ctx) => {
      ctx.ui.notify("Hashline Edit mode active", "info");
    });
  }
}
Key insight: The extensions directory is just a wiring layer. core.ts imports registerReadTool and registerEditTool from src/ and calls them with the pi API. No logic lives here.

The registerTool() shape

Each register*Tool function calls pi.registerTool() with a tool definition object. Here is the full shape, drawn from the edit tool:

ToolDefinition { name: "edit" // unique id; replaces built-in label: "Edit" // UI display name description: EDIT_DESC // LLM-facing tool description promptSnippet: EDIT_PROMPT_SNIPPET // one-liner in tool list promptGuidelines: [...] // only on read (rules for tool usage) parameters: TypeBox schema // validates args before execute() renderShell: "default" // UI wrapper style renderCall(args, theme, context) → Text // live preview while user types renderResult(result, meta, theme, context) → Text // rendered output after execution execute(id, params, signal, onUpdate, ctx) → result // the actual tool logic }

How prompt text is loaded

Tool descriptions live as Markdown files in tool-descriptions/. They are loaded at module load time via readFileSync, not at execution time: 

// From src/edit.ts — runs once when the module is first imported
const EDIT_DESC = readFileSync(
  new URL("../tool-descriptions/edit.md", import.meta.url),
  "utf-8",
).trim();
Why new URL("...", import.meta.url)? This resolves the path relative to the current source file, not the working directory. The path goes up one level from src/ to reach tool-descriptions/. This means the extension works regardless of what directory Pi is started from.

The prompt text pipeline

Three pieces of text feed into the LLM's context for each tool:

tool-descriptions/ │ ├─ read.md → description full tool doc (with {{TEMPLATE}} substitution) ├─ read-snippet.md → promptSnippet one-line summary in tool list └─ read-guidelines.md → promptGuidelines numbered rules embedded in system prompt

The read tool is the only one with guidelines — the read-guidelines.md file contains nine bullet rules parsed into an array and injected into the system prompt. Edit and insert rely on their descriptions alone.

Templates in descriptions

Descriptions can contain {{PLACEHOLDER}} tokens that are replaced at load time. For example, from read.ts: 

const READ_DESC = readFileSync(
  new URL("../tool-descriptions/read.md", import.meta.url),
  "utf-8",
)
  .replaceAll("{{DEFAULT_MAX_LINES}}", String(DEFAULT_MAX_LINES))
  .replaceAll("{{DEFAULT_MAX_BYTES}}", formatSize(DEFAULT_MAX_BYTES))
  .trim();

This means the LLM sees the actual limits ("Default cap: 2000 lines or 50KB") without the source file needing to hardcode numbers in the Markdown.

Parameter schemas (TypeBox)

Each tool defines its parameters with TypeBox — a runtime type system that produces JSON Schema for validation. Here is the insert tool schema: 

const insertEntrySchema = Type.Object({
  anchor: Type.Optional(Type.String({
    description: "3-character hash anchor copied from read output; ...",
  })),
  direction: Type.String({
    enum: ["after", "before", "head", "tail"],
    description: 'Insert direction: "after" or "before" the anchor line, ...',
  }),
  lines: Type.Array(Type.String(), {
    minItems: 1,
    description: "Lines to insert.",
  }),
  current: Type.Optional(Type.String({
    description: "Optional exact current text of the anchor line; ...",
  })),
}, { additionalProperties: false });
Why TypeBox? The description fields inside the schema are exposed to the LLM alongside the JSON Schema — giving it the exact field-level guidance it needs. The additionalProperties: false flag rejects unknown fields, catching typos like anchorr before they reach the logic.

The render lifecycle

Tools have two render phases:

  1. renderCall(args, theme, context) — called before execution, while the user is typing or the LLM is constructing the call. This is where the edit preview appears (showing the path and planned edits). It uses context.state to cache async preview computations so they don't re-run on every keystroke.
  2. renderResult(result, meta, theme, context) — called after execution. For edit/insert this renders the diff with colored +/− lines and summary metrics. It handles partial results (isPartial: true → "Editing..."), errors (red text), and successful changes (diff + line counts).

Both return a Text component from @earendil-works/pi-tui — the terminal UI framework.

The async preview pattern

The edit tool computes a preview (which file, how many blocks) before execution so the user sees what will happen. But reading the file is async — so renderCall fires off a computeEditPreview() promise and stores previewGeneration to ignore stale results: 

// In renderCall — the preview is computed asynchronously
const previewGeneration = (context.state.previewGeneration ?? 0) + 1;
context.state.previewGeneration = previewGeneration;
computeEditPreview(previewInput, context.cwd)
  .then((preview) => {
    // Only apply if still the latest request
    if (
      context.state.argsKey === argsKey &&
      context.state.previewGeneration === previewGeneration
    ) {
      context.state.preview = preview;
      context.invalidate(); // trigger re-render
    }
  });
Why the generation counter? If the user types quickly, multiple preview requests could be in flight. Only the last one should win. The counter lets renderCall detect and discard stale responses.

ExtensionApi capabilities used

The tools use several capabilities from the ExtensionAPI:

  • pi.registerTool() — the core registration method (all extensions).
  • pi.events.emit() — fires "hashline:read-snapshot" after reads so the insert tool can pick up anchors even without calling read itself (insert.ts → registerInsertTool).
  • pi.events.on() — listens for "hashline:read-snapshot" and "hashline:edit-applied" events (insert and undo).
  • pi.on("session_start", ...) — debug notification when PI_HASHLINE_DEBUG=1 (core.ts).

The separation of concerns

extensions/ wiring layer │ default export(pi) → calls registerXxxTool(pi) │ src/*.ts implementation │ registerReadTool(pi) — constructs ToolDefinition, calls pi.registerTool() │ registerEditTool(pi) — constructs ToolDefinition, calls pi.registerTool() │ registerInsertTool(pi) — constructs ToolDefinition, calls pi.registerTool() │ │ Each register function: │ • Reads prompt text from tool-descriptions/ │ • Defines TypeBox parameter schema │ • Implements renderCall / renderResult / execute │ • Calls pi.registerTool() │ tool-descriptions/ prompt text (markdown) read.md, edit.md, insert.md, ...

Why register functions live in src/ not extensions/

The registerXxxTool functions contain substantial logic: schema definitions, render callbacks, and the execute handler. Keeping them in src/ means:

  • Testability — src/ modules can be imported by tests directly, without going through the Pi runtime.
  • Single responsibility — extensions/core.ts just wires; src/edit.ts owns the edit tool definition end-to-end.
  • No duplication — if two extensions needed the same tool (unlikely here), the registration could be called from both without copy-paste.

Compatibility mode

The edit tool supports a legacy compatibility mode controlled by PI_HASHLINE_EDIT_COMPAT=1. When enabled, the schema changes from { path, edits: [...] } to also accept oldText / newText fields from the traditional string-matching edit approach. This is handled by swapping the schema at registration time: 

export function getHashlineEditToolSchema(): typeof hashlineEditToolSchema {
  return isLegacyEditCompatEnabled()
    ? legacyHashlineEditToolSchema
    : hashlineEditToolSchema;
}

// In registerEditTool:
editToolDefinition.parameters = getHashlineEditToolSchema();
pi.registerTool(editToolDefinition);
