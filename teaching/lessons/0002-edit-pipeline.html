0002 — The Edit Pipeline: From Anchor to Atomic Write

Tracing a single edit() call through every stage until the bytes hit disk

The big picture

An edit() call travels through four major phases. Here they are, with the owning source file for each:

edit() call arrives │ ├─ 1. Validate & Normalize edit.ts │ assertEditRequest() → normalizeEditItems() │ ├─ 2. Resolve & Apply mutation.ts │ applyMutation() │ ├── resolveEditTarget() read + normalize file │ ├── buildHashlineFile() compute hash array │ ├── resolveEditAnchors() parse anchor strings │ ├── 3-tier match exact → fuzzy → merge │ ├── resolveEditSpans() map to byte-level spans │ └── applySpans() splice content │ ├─ 3. Write to Disk fs-write.ts │ writeFileAtomically() │ └─ 4. Build Response edit-response.ts generateDiffString() → text + details

Phase 1: Validate & Normalize

Before touching the file, the edit tool validates the request shape. assertEditRequest() in edit.ts checks that path is a non-empty string and edits is a non-empty array, with each entry having non-empty start and end strings and a lines array.

Then normalizeEditItems() converts the raw JSON into typed HashlineToolEdit[] objects — each with op: "replace", a pos (the start hash), end, and lines:

// Raw JSON from the model: { start: "aB3", end: "aB3", lines: ["new content"] } // → normalizeEditItems() produces: { op: "replace", pos: "aB3", end: "aB3", lines: ["new content"] }

Phase 2: Resolve & Apply (the engine room)

applyMutation() in mutation.ts is the heart of the system. It orchestrates every step from file reading through span application. Let’s trace through each sub-step.

2a. Read the file

resolveEditTarget() reads the file, checks it exists and is writable, strips any BOM, normalizes CRLF→LF, and verifies it’s not a binary or directory. The result is a normalized string plus the original line ending and BOM — preserved for round-trip fidelity.

2b. Build the hash array

buildHashlineFile(content) splits the normalized text into visible lines and calls computeLineHashes() to produce the single canonical hash array. It also precomputes lineStarts[] — byte offsets for each line — which is critical for the span computation later.

2c. Resolve anchor strings into positioned edits

resolveEditAnchors(toolEdits, currentFile) in hashline.ts does two things:

  1. Parse anchor references. Each pos/end string is validated against HASH_RE (exactly 3 base64url chars). Anything else — line numbers, content, separators — triggers diagnoseLineRef() which returns a specific error message.
  2. Normalize lines. hashlineParseText() converts the model’s lines (which may arrive as a string or array) into a clean string array. Crucially, it also calls assertNoDisplayPrefixes() — rejecting any line that starts with a hashline prefix like aB3│content. The model must send literal content, not rendered read output.
Architecture guardrail: AGENTS.md states: “Do not introduce autocorrection heuristics into applyHashlineEdits. The runtime must not silently patch model errors.” If the model sends a hashline prefix, the edit is rejected, not silently stripped.

2d. Match anchors to live lines

You already learned the 3-tier resolution from lesson 0001. Here’s how it fits in the pipeline:

  1. partitionExact — tries to resolve each anchor uniquely in the current file.
  2. fuzzyMatch — for anchors that didn’t match, tries hash-only relocation with a [W_RELOCATED] warning.
  3. threeWayMerge — if anchors still don’t match, checks the snapshot from the last read. If they match there, rebases the edit onto current content.

At the end, every edit either has a known line position — or the whole request fails with a [E_STALE_ANCHOR] or [E_AMBIGUOUS_ANCHOR] error.

2e. Resolve edits into byte-level spans

resolveEditSpans(file, edits) is where the abstract edit operations become concrete byte-level instructions. For each edit it produces an EditSpan:

// EditSpan — the byte-level instruction: { index: 0, // which edit in the request label: "replace aB3", // human-readable label start: 247, // byte offset where replacement begins end: 268, // byte offset where replacement ends replacement: "new\nlines", // the spliced-in content op: "replace", // operation type lineShiftStart: 14, // line where shifts begin (for downstream spans) lineDelta: 1 // net line count change (±) }

How start/end byte offsets are computed: The precomputed lineStarts[] array from buildHashlineFile() maps line numbers to byte positions. Given a resolved line n:

  • Single-line replace: start = lineStarts[n-1] (first byte of line n), end = lineStarts[n-1] + lines[n-1].length (last byte of line n).
  • Multi-line replace: start at first byte of the first line, end at last byte of the last line. The newline between lines n and n+1 is included in the deleted range.
  • Delete (lines = []): Same as replace but replacement = "". Special case: if deleting the last line, start is nudged back one byte to also consume the preceding newline.

Append and prepend work differently — they insert pure content without deleting anything. Their spans have start === end (zero-length range) and the replacement includes the necessary newline:

// Append after line N: start = lineStarts[N] // right after line N's \n end = lineStarts[N] // zero-length replacement = "new line\n" // includes trailing \n // Prepend before line N: start = lineStarts[N-1] // first byte of line N end = lineStarts[N-1] // zero-length replacement = "new line\n" // push line N down with trailing \n
Insert head/tail are handled even more simply: their spans target byte 0 or byte content.length with a zero-length range.

Detecting problems during span resolution

resolveEditSpans doesn’t just compute spans — it also detects several important edge cases:

  • Noop detection. If the replacement text exactly matches the original lines, the edit is classified as a noop and excluded from spans. The response reports Classification: noop and no file write occurs.
  • Boundary duplication warnings. If the replacement’s first line matches the preceding surviving line (or the last line matches the next surviving line), a warning is emitted. This catches cases where the model accidentally duplicates a line that’s about to survive the edit.
  • Overlap detection. If two edits in the same request touch overlapping byte ranges, the entire request is rejected with [E_EDIT_CONFLICT]. The model must merge them first.
  • Current mismatch. If the model supplies current and it doesn’t match the live line, the edit is rejected with [E_CURRENT_MISMATCH].

2f. Apply spans to content

applySpans(file, spans) is deceptively simple. It takes the spans, sorts them in reverse-end order (processing from the end of the file toward the beginning), and splices each replacement into the content string:

// src/hashline.ts — applySpans const orderedSpans = [...spans].sort((left, right) => { if (right.end !== left.end) return right.end - left.end; return left.index - right.index; }); let result = file.content; for (const span of orderedSpans) { result = result.slice(0, span.start) + span.replacement + result.slice(span.end); }

Why reverse-end order? It keeps byte offsets stable. If you edit from the top first, every downstream span’s offsets become invalid because the text shifts. By editing from the bottom up, each span’s start/end still indexes into the original content — no recalculation needed.

Phase 3: Atomic Write

writeFileAtomically() in fs-write.ts implements three safety guarantees:

  1. Symlink-safe path resolution. resolveMutationTargetPath() walks the path component-by-component, resolving symlinks. Handles ELOOP (symlink cycles) and partial existence (some parts of the path may not exist yet).
  2. Hardlink detection. If the target file has nlink > 1 (multiple hardlinks pointing to the same inode), the atomic write strategy is skipped — writing through the temp-file + rename would break the hardlink. Instead, the file is overwritten in-place.
  3. Temp-file + rename. For normal files: write to .tmp-<uuid> with flag: "wx" (exclusive create — fails if the temp already exists), preserving the original mode. Then rename(temp, target). On POSIX systems, rename() is atomic: any reader sees either the complete old file or the complete new file, never a partial write.
Why this matters: Without atomic writes, a crash or concurrent reader could see a half-written file. The temp + rename strategy guarantees that the file contents are always complete and consistent.

Phase 4: Build Response

After the write succeeds, buildChangedResponse() in edit-response.ts produces the output the model sees. It does two things in parallel:

  • Model-visible text: A compact summary — “Applied changes to path”, optionally “Fresh anchors for follow-up edits” (up to 12 anchor+content pairs from the diff), and any warnings.
  • Host-only details: The full unified diff (with hashline anchors on + and context lines, using the new file’s hash array), snapshot ID, and metrics (lines added/removed, noop count, warnings count). These are rendered by the UI shell but are not part of the model’s context window — saving tokens while giving the human a rich diff view.

Putting it all together

Here’s the complete trace of a single-line edit, annotated with the exact source locations:

edit({ path: "src/main.ts", edits: [{ start: "aB3", end: "aB3", lines: ["fixed"] }] }) 1. assertEditRequest() edit.ts: confirms path/edits shape 2. normalizeEditItems() edit.ts: → [{ op:"replace", pos:"aB3", end:"aB3", lines:["fixed"] }] 3. applyMutation() mutation.ts: enters file mutation queue a. resolveEditTarget() edit.ts: → normalized "import { foo }\n..." b. buildHashlineFile() hashline.ts: → { lines, lineHashes, lineStarts, content } c. resolveEditAnchors() hashline.ts: → [{ op:"replace", pos:{hash:"aB3"}, lines:["fixed"] }] d. partitionExact() fuzzy-match.ts: aB3 → line 3 in current file ✓ e. resolveEditSpans() hashline.ts: → [{ start:42, end:55, replacement:"fixed", op:"replace" }] f. applySpans() hashline.ts: result = "...".slice(0,42) + "fixed" + "...".slice(55) 4. writeFileAtomically() fs-write.ts: tmp file → rename to target 5. buildChangedResponse() edit-response.ts: "Applied changes to src/main.ts\n..."

Check your understanding

Q1. Why are spans applied in reverse-end order?
✓ Correct! By processing spans from the end of the file first, earlier spans' byte offsets remain valid because the text after them hasn't been modified yet.
✗ Not quite. The reason is about keeping byte offsets valid. Since spans are computed against the original file content, editing from the bottom up means each span's start/end offsets still point to the right bytes — the text hasn't shifted yet.
Q2. What happens when two edits in the same call touch overlapping line ranges?
✓ Correct! resolveEditSpans detects overlapping byte ranges and rejects the whole request. The model must merge the edits into one non-overlapping change or split them into separate requests.
✗ Not quite. The system explicitly checks for overlaps and rejects the request with [E_EDIT_CONFLICT]. The philosophy is: don't silently resolve ambiguity — make the model fix it.
Q3. Why does writeFileAtomically fall back to in-place write for hardlinked files?
✓ Correct! With a hardlink, rename() replaces only one directory entry. The other hardlinks would still point to the old (unedited) inode — so the file would appear unchanged from those paths. In-place write keeps all hardlinks pointing to the same data.
✗ Not quite. The issue is that rename() replaces a single directory entry. If the file has multiple hardlinks, the other links would still reference the old inode with the original content — making the edit appear to have no effect from those paths.
