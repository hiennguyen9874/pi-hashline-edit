0001 — The Hashline Anchor System

Why LINE#HASH│content, and how the anchors survive edits

The problem: line numbers rot

Most editing tools reference lines by number: “replace line 42” or “delete lines 10–15.” This works fine for a human sitting at an editor — but an LLM issuing an edit cannot know whether the file has changed since it last read it. Another tool call, another agent, or even a background process might have shifted every line by the time the edit executes.

The hashline protocol solves this with content-addressed anchors. Instead of line 42, the model says “replace the line with hash aB3.” The hash is computed from the line’s content — and its neighbors — so it stays valid even if the line moves up or down.

Key insight: Anchors are stable under line insertions/deletions. If someone adds 5 lines above line 42, a line-number reference breaks; a hash anchor still finds the right line because the hash hasn’t changed.

How a hash is computed

Each hash is exactly 3 characters from a base64url alphabet: A–Z a–z 0–9 - _. That’s 64³ = 262,144 possible values.

The hash is context-aware: it’s computed from the line itself and both its neighbors (prev, curr, next). This prevents unrelated lines from accidentally sharing the same hash, which would make anchors ambiguous.

// src/hash-format.ts — the core hash function function computeLineHashFromContext(prev, curr, next, retry = 0) { const canonical = `${prev}\0${curr}\0${next}`; const input = retry === 0 ? canonical : `${canonical}\0R${retry}`; return hashToString(xxhash.h32(input, 0) >>> 0); }

Three details matter here:

  1. Context anchoring. Hashing prev\0curr\0next ties the anchor to its neighborhood. Two identical lines in different contexts get different hashes. This is why the protocol can distinguish nearly-identical lines in a file full of } and return;.
  2. Collision resolution. The retry parameter exists because even with context, hash collisions can happen within a single file. When computing the full hash array for a file, computeLineHashes in hash-format.ts retries with incrementing suffixes (:R1, :R2, …) until every hash is unique within that file.
  3. One perfect hash array per file. The file has exactly one canonical hash array, computed by buildHashlineFile() in hashline.ts. Every tool (read, edit, insert) uses this same computation — hashes are never recomputed differently per call site.

The display format

When you call read, each line appears as:

42#aB3│import { foo } from './lib';

The parts:

  • 42 — a display-only line number, for navigation with offset/limit.
  • # — the anchor separator (ANCHOR_SEP in anchor-display.ts).
  • aB3 — the 3-character content hash. This is what you copy into edit or insert.
  • │ — the content separator (CONTENT_SEP).
Critical rule: Line-qualified anchors (42#aB3) are rejected by edit and insert. Copy only the 3-character hash. The line number is display-only — including it triggers [E_BAD_REF] from diagnoseLineRef() in hashline.ts.

Anchor resolution: three tiers

When an edit arrives with hash anchors, the runtime doesn’t just look up the hash and call it done. It uses a 3-tier resolution strategy in mutation.ts:

  1. Tier 1 — Exact match (partitionExact): The hash resolves uniquely in the current file, and the anchor’s optional line position hasn’t moved. Fast path.
  2. Tier 2 — Fuzzy relocation (fuzzyMatch): The hash resolves uniquely but at a different line. The file changed since the read. The edit is relocated with a [RELOCATED] warning.
  3. Tier 3 — Three-way merge (threeWayMerge in merge.ts): The hash doesn’t match the current file at all. The runtime checks the snapshot from the last read. If the anchors match the snapshot, it applies edits to the snapshot, diffs old→edited, and applies that diff to the current file. Like git rebase.

If none of these work, you get a [E_STALE_ANCHOR] or [E_AMBIGUOUS_ANCHOR] error telling you to re-read the file.

Why this design?

The invariant driving all of this is stated in AGENTS.md:

Hashes are computed through one perfect per-file hash array; do not recompute call-site-specific anchors.

This single-source-of-truth property means that every tool sees the same hashes for the same file content. There’s no drift between what read shows and what edit accepts. And because hashes are context-aware, collision-resolved, and display-only line numbered, the protocol stays robust under concurrent edits, paging, and truncation.

Check your understanding

Q1. If you insert 3 lines at the top of a file, what happens to the hashes of lines below?
✓ Correct! Hashes are computed from content (and neighbors), not position. Inserting lines above doesn't change the content or neighbors of lines below — their hashes remain valid.
✗ Not quite. Since hashes are computed from line content and its neighbors, inserting lines above doesn't affect lines below. Their content hasn't changed, and their neighbors haven't changed. The hashes stay the same.
Q2. What does the retry parameter in computeLineHashFromContext do?
✓ Correct! The retry mechanism ensures every hash in a file is unique by appending disambiguation suffixes when collisions occur.
✗ Not quite. The retry parameter resolves hash collisions within a single file. If two different lines produce the same 3-char hash, the engine retries with :R1, :R2, etc. appended to the input, changing the hash output until it's unique.
